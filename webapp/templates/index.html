What is a Pattern?

Patterns are deterministic rule sets that shape the Iterate Card. They don't use scoring and can be combined by comma.

  • constraint-ledger - Keep a running list of constraints (include/avoid/not_too/vibes); echo and confirm before action. Prevents contradictions and preserves user intent.
  • contrastive-clarify - Ask one either/or to pin down an ambiguous term; reflect the choice; proceed. Reduces guessing and keeps momentum.
  • exemplar-propose - Offer two tiny, concrete options that both fit constraints; ask which is closer or what to tweak. Anchors vague requests with concrete proposals.
  • override-hook - Simple staff commands (override/lock/reduce/reset) that apply immediately and are echoed. Enables fast corrections without restarting the flow.
  • state-bag - Keep explicit state (goal/include/avoid/not_too/memory/next_step/confirmed); echo updates; confirm before finalizing. Prevents drift and contradictions.
  • slot-filling - Ask only for missing required fields; never re-ask captured ones; echo and confirm summary. Reduces redundant questions.

Tip: Leave blank to auto-detect based on your Friction. Combine like constraint-ledger,contrastive-clarify. Learn more: Why These Patterns?

Why three modes?
  • Iterate Card (Fix) — One atomic fix today: Diagnosis → Fix → Prompt patch → Validation.
  • Plan (Explain) — Align stakeholders: Context → Objective → Flow → Reasoning Path → Output.
  • Ticket (Handoff) — Shareable brief: Client → Problem → Test → Goal → Success.

Pick Iterate Card for fast copy‑paste. Switch to Plan or Ticket when sharing.
